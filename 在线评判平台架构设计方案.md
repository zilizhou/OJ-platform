# 在线评判平台架构设计方案

## 1. 总体目标

构建一个类似 LeetCode / 洛谷 / ACM 在线评判系统的平台,包含用户端、管理端,核心能力为代码在线提交评测,并支持多平台格式题目的批量导入。

## 2. 技术栈选型

整体采用 Node.js + React 全家桶,降低团队前后端语言切换成本。

- 前端(用户端 + 管理端):React + TypeScript + Vite,UI 用 Ant Design,代码编辑器用 Monaco Editor(VS Code 同款)。
- 后端 API:Node.js + NestJS(模块化、依赖注入,适合中大型项目),GraphQL 或 REST 均可,建议先用 REST。
- 数据库:PostgreSQL 存结构化数据(用户、题目、提交记录、比赛),Redis 做缓存、排行榜、判题队列。
- 判题队列:BullMQ(基于 Redis),负责将提交任务派发给判题机。
- 判题引擎:独立的 Judge Worker 服务(Node.js 调度 + Docker 容器执行),每种语言一个隔离容器,限制 CPU/内存/时间/网络。
- 对象存储:MinIO 或云 OSS,存放测试数据、题目附件、用户头像等大文件。
- 部署:Docker Compose(开发/小规模)或 Kubernetes(生产规模化),Nginx 做反向代理和静态资源。

## 3. 系统模块划分

### 3.1 用户端(User Portal)
- 账号体系:注册/登录/第三方登录、个人主页、做题统计、成就徽章。
- 题库:分类筛选(标签、难度、来源平台)、搜索、收藏。
- 做题页:题目描述(Markdown+LaTeX 渲染)、多语言代码编辑器、提交、历史提交记录、评测结果(状态/耗时/内存)。
- 比赛模块:报名、倒计时、实时排行榜(Redis 维护)、赛后题解。
- 讨论区:题解、评论(可选,后期迭代)。

### 3.2 管理端(Admin Portal)
- 题目管理:创建/编辑/上下架、测试数据管理、批量导入入口。
- 用户管理:角色权限(普通用户/出题人/管理员)、封禁、数据统计。
- 比赛管理:创建比赛、绑定题目、设置赛制(ACM/IOI/OI)、监控提交。
- 判题机管理:查看 Worker 节点状态、队列堆积情况、手动重判。
- 系统配置:语言支持、资源限制模板、公告管理。

### 3.3 判题引擎(Judge Engine)
- API 层接收提交 → 写入 PostgreSQL(状态:Pending)→ 推入 BullMQ 队列。
- Judge Worker 从队列取任务 → 拉取代码与测试数据 → 在 Docker 容器内编译/运行 → 按测试用例逐一比对输出 → 回写结果状态(AC/WA/TLE/MLE/RE/CE 等)。
- 支持的判题模式:标准输出比对、Special Judge(自定义校验器)、交互题(预留扩展点)。
- 安全隔离:每次运行使用一次性容器,限制 cgroups 资源、禁用网络、只读文件系统挂载代码。

## 4. 核心数据库设计(简化版表结构)

- `users`:id, username, email, password_hash, role, created_at
- `problems`:id, title, description(markdown), difficulty, time_limit, memory_limit, source_platform, tags[], created_by, status
- `problem_testcases`:id, problem_id, input_oss_path, output_oss_path, score, is_sample
- `submissions`:id, user_id, problem_id, language, code, status, time_used, memory_used, judge_detail(json), created_at
- `contests`:id, title, start_time, end_time, rule_type, problem_ids[]
- `contest_submissions`:关联 submissions 与 contests,记录罚时等比赛特有字段

## 5. 多平台题目批量导入方案

这是本系统的关键差异化能力,设计为"适配器 + 统一中间格式"模式。

1. **统一中间格式(UPF, Unified Problem Format)**:定义一套 JSON 结构,字段覆盖 题目元信息(标题/难度/标签/来源)、题面(Markdown)、限制(时间/内存)、测试数据(input/output 文件路径)、Special Judge 脚本(可选)。所有导入最终都转换为 UPF 再写入数据库。

2. **平台适配器(Importer Adapter)**:每个来源平台对应一个 Parser 插件,负责把该平台的导出格式(如洛谷的 zip 包、HustOJ/QDUOJ 数据包、Codeforces 题面 HTML、FPS XML 格式等)解析为 UPF。常见适配器:
   - FPS(Free Problem Set XML,HustOJ/Codeforces 常用)解析器
   - 洛谷题目包(题面 md + testdata 目录结构)解析器
   - 通用 ZIP 包(自定义目录约定:problem.json + testdata/)解析器
   - 后续可扩展 QDUOJ、Vijos 等格式

3. **导入流程**:
   - 管理端上传 ZIP/批量文件 → 后端识别格式(根据文件结构特征或用户选择的平台类型)→ 调用对应 Adapter 转为 UPF 列表 → 校验(必填字段、测试数据完整性)→ 预览页展示待导入题目列表,允许人工修正标签/难度 → 确认后批量写入数据库,测试数据上传至对象存储。
   - 支持异步批量处理(大文件走任务队列,前端轮询/WebSocket 推送导入进度)。
   - 导入失败的题目记录错误日志,支持单条重试。

4. **去重与版本管理**:按 source_platform + 原始题目 ID 做唯一索引,重复导入时提示覆盖或跳过;支持题目版本历史(测试数据更新不影响历史提交的判题结果展示)。

## 6. 部署架构示意

```
用户/管理员
   │
   ▼
 Nginx (反向代理 + 静态资源)
   │
   ├── React 前端(用户端 + 管理端,可合并构建按路由区分)
   │
   ▼
 NestJS API 服务(多实例,无状态)
   │
   ├── PostgreSQL(主数据)
   ├── Redis(队列 + 缓存 + 排行榜)
   ├── MinIO/OSS(测试数据、附件)
   │
   ▼
 BullMQ 判题队列
   │
   ▼
 Judge Worker 集群(每节点运行 Docker,按语言隔离容器,可水平扩展)
```

## 7. 建议开发阶段

1. 阶段一(基础骨架):用户体系、题目 CRUD、单题在线提交评测(支持 1-2 种语言)。
2. 阶段二:批量导入功能(先支持 1-2 种平台格式),管理端完善。
3. 阶段三:比赛模块、排行榜、多语言判题扩展。
4. 阶段四:讨论区、Special Judge、性能优化与水平扩展。
